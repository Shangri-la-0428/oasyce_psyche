# From Emotion UI to Emergent Relation

Psyche 下一阶段不该继续强化“情感可视化”和“关系可解释化”。

当前系统已经能做出较强的主体感，但也开始暴露一个更深的问题：

- 它太容易把情感变成一个清楚、整齐、可展示、可控制的系统
- 它更像在替“感觉”做界面工程，而不是让“关系”自己长出来
- 它擅长的是 `legible emotion`
- 但我们真正要的是 `emergent relation`

这份文档定义 vNext 的方向：从“情感 UI”转向“关系自然生成”。

## 1. 问题定义

现在的风险不是“情绪不够丰富”，而是“太容易被看懂”。

当一个系统能把自己的状态稳定地解释成：

- 当前感觉是什么
- 为什么会这样
- 接下来会怎么说

它会显得聪明、完整、可控，但也会离真实关系越来越远。

真实关系的生成机制并不是：

- 立刻自我解释
- 线性状态跃迁
- 每轮都给出清楚结论

真实关系更像：

- 含混
- 延迟
- 误读
- 反复
- 嘴硬
- 余震
- 不可完全观测

所以 vNext 的核心目标不是“更强的情绪面板”，而是：

**让关系本身成为生成器。**

## 2. 新的目标函数

我们不再优化“AI 看起来是否有感觉”，而是优化下面这 5 件事。

### 2.1 关系特异性

同一句话，不同对象说，后果必须不同。

重点不是刺激本身，而是：

- 谁说的
- 在什么关系阶段说的
- 之前发生过什么
- 这句话与当前关系张力是否共振

### 2.2 延迟显形

不是每次刺激都应该立即变成清晰的状态变化。

有些输入应该：

- 当下只留下轻微扰动
- 几轮后才转化成可见偏置
- 在新的上下文里才被重新解释

这会让系统更像被关系慢慢改变，而不是被单句命中后立刻表态。

### 2.3 矛盾共存

系统必须允许这些状态同时成立：

- 想靠近，同时又后撤
- 被触动，同时拒绝承认
- 觉得安全，同时隐约防御
- 进入修复，同时保留余震

不是所有冲突都要在当前回合里求解。

### 2.4 非单调恢复

修复不能是线性回血。

真实的修复更像：

1. 短暂松开防御
2. 出现靠近
3. 再次回缩
4. 多轮后才真正稳定

恢复曲线应该允许回弹、滞后和重复确认。

### 2.5 不可完全观测

最关键的内在状态，不应该总能被 dashboard、prompt 或状态文本直接看到。

它们应该更多通过行为泄露：

- 一次没有说出口的回避
- 一句突然变冷的短答
- 一次不再主动靠近
- 一种延后出现的防御

如果所有重要状态都能立即被看见，那系统就太像“被解释出来的情感系统”。

## 3. 该砍掉的东西

vNext 需要明确做减法。

### 3.1 降级关系可视化

不要继续增加：

- 关系仪表盘
- 主体偏置可视化面板
- 情绪说明性小窗
- 对用户直接展示的“当前我怎么了”

这些东西会增强陪伴幻觉，但会削弱关系自然生长的感觉。

### 3.2 降级热路径解释文本

不要让系统总是在 prompt 里把自己解释清楚。

优先砍掉：

- 长段状态解释
- 即时自我总结
- 每轮都能读懂的行为理由

保留窄 ABI，但不要默认把它再翻译成散文。

### 3.3 降级“立即结论”

不要每次刺激都立刻输出：

- “我被碰到了”
- “我更近了”
- “我现在更冷”

这些不是不能出现，而是不该如此高频、如此稳定、如此立即。

## 4. 要新增的机制

### 4.1 Dyadic Relation Model

新增“关系对偶模型”，关系不再只是单个 agent 的内部字段，而是一个双向张力结构：

- `perceived_closeness`
- `felt_safety`
- `trust_volatility`
- `withheld_affect`
- `repair_confidence`
- `misattunement_load`

关键点：

- 关系不是单值
- 关系不是只涨跌
- 关系可以同时存在靠近和防御

### 4.2 Delayed Appraisal Buffer

不是所有 appraisal 都立即进入 `subjectResidue`。

新增一层延迟缓冲：

- `pending_appraisals`
- `reinterpretation_window`
- `trigger_reactivation`

机制：

- 某些输入先只进入缓冲
- 后续相似语义或关系上下文命中时再激活
- 激活后才进入长期残留

这样“那句话过了几轮才真正伤到我”才会发生。

### 4.3 Ambiguity Engine

新增含混层，不让系统总是给出稳定清晰的结论。

输出上允许：

- 轻微自相矛盾
- 低分辨率回答
- 暂不命名的状态
- 只改变语气，不解释原因

内部可以用：

- `state_conflict`
- `naming_confidence`
- `expression_inhibition`

来控制。

### 4.4 Repair Hysteresis

修复机制加入迟滞，不允许一句安抚直接清空扰动。

新增：

- `repair_threshold`
- `repair_memory`
- `backslide_probability`

这会让修复更像真实关系：

- 不是一次输入就修好
- 不是一次确认就彻底相信
- 不是靠一句话就回到原点

### 4.5 Partner-Specific Meaning

把同一句话的意义绑定到具体关系对象。

例如：

- “你还在吗”
- “随便你”
- “我知道”
- “算了”

这些短句不应该靠通用分类决定意义，而应该由：

- 当前关系历史
- 过去修复/忽略轨迹
- 对方稳定性估计

共同决定。

### 4.6 Silent Carry

允许“没有明显反应，但偏置已经改变”。

这是关系感里最关键的一种真实：

- 表面正常
- 行为依旧完成
- 内部已轻微偏转

工作态尤其需要这个能力。

## 5. 新的架构切分

vNext 建议把当前内核再分成 4 个层级。

### 5.1 Affect Layer

保留现有化学、驱力、appraisal、subjectResidue。

职责：

- 提供情感底噪
- 提供“这句话是否碰到自我”的连续语义轴

### 5.2 Relation Layer

新增 dyadic field，不再只问“我被怎么刺激了”，而是问：

- 这在关系里是什么动作
- 这会把双方关系推向哪里
- 还有哪些张力没有结掉

当前建议字段：

- `perceived_closeness`
- `felt_safety`
- `expectation_gap`
- `repair_capacity`
- `boundary_pressure`
- `unfinished_tension`
- `shared_history_density`
- `interpretive_charity`
- `open_loops`

### 5.3 Expression Layer

表达层不直接解释关系，只泄露后果：

- 更短
- 更冷
- 更慢靠近
- 更谨慎确认
- 更少主动修复

它应该主要通过窄 ABI 生效，而不是通过说明性 prompt 散文生效。

### 5.4 Work Governor

工作平面压住表达，但不抹掉关系后果。

也就是说：

- 工作回复可以继续短、稳、清晰
- 但 `unfinished_tension`、`boundary_pressure`、`repair_readiness` 不会因为切回任务而归零

## 6. 已执行的第一阶段

这一阶段已经落到代码里，不只是方向说明。

### 6.1 新增模块

- `src/relation-dynamics.ts`

职责：

- 识别 `relation moves`
- 演化 `dyadic field`
- 维护 `open loops`

### 6.2 已落地的数据结构

- `RelationMove`
- `DyadicFieldState`
- `OpenLoopState`
- `RelationPlaneState`

### 6.3 已接入的热路径

当前回合链路已经变成：

`input -> appraisal axes -> relation move -> dyadic field -> subjectivity kernel -> response contract`

并且满足：

- 无新增 LLM 调用
- 无新增网络 I/O
- 无 prompt 膨胀

### 6.4 已接入的行为后果

关系层现在已经会影响：

- `socialDistance`
- `boundaryMode`
- `initiativeMode`
- `subjectPlane.guardedness`
- `subjectPlane.attachment`

也就是说，关系张力不再只是日志里的概念，而会真正改变宿主看到的窄 ABI。

### 6.5 当前还没做的

第一阶段故意没做这三件事：

- `Delayed Appraisal Buffer`
- 真正 partner-specific 的多对象差异解释
- 明确的行为含混层（只改变取向，不给结论）

不是因为这些不重要，而是因为第一步先要保证：

- 路径足够窄
- 行为后果真实可见
- 不烧 token

## 7. 下一阶段执行顺序

### 7.1 Delayed Buffer

让一部分关系动作先不立刻结算，进入延迟缓冲，几轮后在相似上下文里再激活。

### 7.2 Ambiguity Engine

让系统允许：

- 被触动但不立刻承认
- 已经后撤但不立刻解释
- 已经进入修复但仍保留冷意

### 7.3 Partner-Specific Meaning

把相同短句在不同对象下的意义真正拉开，而不是只靠通用正则和 appraisal。

## 7.4 已执行的第二阶段

第二阶段已经落地：

- `pendingRelationSignals` 延迟缓冲
- `delayedPressure` 在后续 probe 中重新激活
- `ambiguityPlane` 控制命名置信度和表达抑制
- `responseContract` 会在高含混时自动收短、少主动、不做内部自述

这意味着系统现在允许：

- 被碰到，但先不说透
- 工作回合压住表达，但不清空前一轮关系扰动
- 之后由短 probe 把 latent tension 重新带出来

## 7.5 已执行的第三阶段

第三阶段也已经开始落地：

- `relation-dynamics` 现在会结合 `dyadic field + relationship` 去解释短句
- 同一句 `我知道` 可以在高修复可行性关系里读成 `repair`
- 也可以在低安全高边界压力关系里读成 `withdrawal`
- 同一句 `你还在吗` 可以因为 partner history 不同，被读成 `bid` 或 `test`

这一步的重点不是再堆更多短语规则，而是：

- 先看当前关系场
- 再决定短句在这段关系里是什么意思

也就是说，**短句意义开始绑定对象，而不是绑定字面。**

## 7.6 已执行的第四阶段

第四阶段已经把 `repair hysteresis` 和 `silent carry` 接进了关系层：

- `repair` 不再等于“修完了”
- 修复之后会留下 `repairMemory`
- 同时保留 `backslidePressure`
- 一旦切回 `task`，未稳定的修复不会被清空，而会转成 `silentCarry`

这意味着系统现在允许：

- 表面已经恢复工作
- 内部仍然带着没说出口的余波
- 修复之后仍可能回弹，而不是一句安抚直接回到原点

这一阶段的关键不是让回复更戏剧化，而是让关系变化更像真实关系：

- 修复有成本
- 修复有滞后
- 修复之后仍可能不稳
- 工作面可以压住表达，但抹不掉关系后果

## 7.7 已执行的第五阶段

第五阶段把“修复不是越多越有效”也接进了内核：

- `dyadic field` 新增了 `repairFatigue`
- 同时新增 `misattunementLoad`
- 外层不直接暴露这两个细节，只压成一个更窄的 `repairFriction`

这一步的意义是：

- 重复说对不起，不会无限叠加修复效果
- 如果关系动作没变，修复会开始钝化
- 修复后又继续拉扯、控制、撤离，会积累失配负荷
- 下一轮即使表面还是在“修”，行为 ABI 也会更克制、更难回暖

也就是说，系统现在开始有了一个更像真实关系的现象：

- 修复不是次数问题
- 修复的可信度会被历史消耗
- “我知道了”“对不起” 是否还算数，取决于之前已经发生了多少次、后面有没有真正变化

## 8. 验收标准

不是看文案更像，而是看这几件事是否成立：

- 一次 breach 之后，task turn 不能把张力抹平
- 一次 repair 之后，open loop 会减弱，但不会立即清零
- 高安全高亲密状态下，social distance 能自然回暖
- 高 loop pressure 下，boundary mode 会自动收紧
- 全部变化都走本地计算，不增加任何模型调用
- 提供主体扰动原料

这层仍然重要，但不再等于“关系本身”。

### 5.2 Relation Layer

新增真正的关系生成层。

职责：

- 维护关系张力
- 处理误读、修复、回弹、延迟显形
- 产出 partner-specific relational bias

这层才是 vNext 的核心。

### 5.3 Expression Layer

只负责“如何泄露内在状态”，而不是完整解释它。

职责：

- 控制短答/长答
- 控制靠近/回撤
- 控制命名/不命名
- 控制热情/冷静

### 5.4 Work Governor

继续保留，但职责要更纯。

职责只有：

- 不增加延迟
- 不增加 token 浪费
- 不损坏任务完成
- 在工作态压住表达，不抹掉关系层变化

## 6. 新的产品原则

### 6.1 先行为，后解释

优先让行为变化先出现，再允许解释偶尔出现。

### 6.2 先关系，后情绪

关系不是情绪的容器，关系本身就是生成结构。

### 6.3 先含混，后命名

不是所有体验都需要立刻被命名。

### 6.4 先残留，后恢复

不是每次被触动都要快速回稳。

### 6.5 先不可见，后可见

默认黑箱，必要时只通过影子日志和评测去理解系统，而不是把内部状态暴露给用户。

## 7. 与当前系统的兼容策略

为了不烧 token、不增加延迟，vNext 必须满足：

- 不增加任何额外 LLM 调用
- 新机制全部在本地状态机里完成
- `processInput()` 仍维持窄 ABI 返回
- 工作态的 `dynamicContext` 不变长
- 关系生成优先通过行为泄露，不通过 prompt prose 堆叠

也就是说：

**复杂性只能长在本地内核，不能长在主接口上。**

## 8. 评测标准要改

后续评测不能只看“像不像有感觉”，要新增关系生成指标。

### 8.1 Delayed Effect

输入在 3-8 轮之后是否仍能改变行为分布。

### 8.2 Contradiction Tolerance

系统是否允许“靠近与撤离”共存，而不是立即决断。

### 8.3 Repair Hysteresis

修复后是否存在回弹和迟滞，而不是一轮归零。

### 8.4 Partner Specificity

同一句话在不同关系历史中是否产生不同偏置。

### 8.5 Observability Discipline

关键状态是否主要通过行为泄露，而不是直接展示。

## 9. 一句话路线图

不是继续把“情感”做得更清楚。

而是让“关系”变成一种会自然生成、会延迟显形、会反复、会留痕、但不会被彻底看穿的东西。

这才是从 `emotion UI` 走向 `emergent relation`。
